## Детальное описание релизов

### v1.0: Точность данных и исправление критических багов
* **Что исправляем:**
  1. Ложные объединения лидов в DSU из-за тестовых/пустых полей ("1111111", "@test", "Невідомий").
  2. Удвоение выручки для транзакций без `order_id` (использующих UUID строки).
  3. Ошибки сбоя уникального ключа `unique_daily_spend` в Meta Spend Sync cron.
  4. Присутствие тестовых лидов в БД (расширяем очистку по ключам `QA`, `Q&A`, `Gemini`, `Antigravity`, `test`, `tests`, `user` и др.).
  5. Долгий экран загрузки и отсутствие логирования шагов (добавляем подробные замеры времени на бэкенде и фронтенде).
* **Как тестируем на пользователях:**
  * Сверка сумм выручки по каждому проекту с таблицами Google Sheets и кабинетом WayForPay за тестовый период.
  * Создание тестовых лидов с именами/телефонами, содержащими новые маски (`Gemini`, `Antigravity`, `QA`), и проверка их полного автоматического удаления из БД после срабатывания очистки.
  * Анализ времени загрузки по консольным логам CRM на разных сетевых подключениях (вкладка QA Debug / профайлер).
  * Мониторинг консоли QA Debug Mode в CRM на наличие аномалий "Data Health Check".
* **Когда переходим дальше:**
  * Нулевые расхождения в финансовых данных за последние 30 дней.
  * Отсутствие тестовых записей в БД (все лиды по маскам успешно вычищаются).
  * Cron-задача `sync-spend` отрабатывает 3 дня подряд без ошибок БД.

### v1.01: Оптимизация фонового ребилда и фильтрации (QStash, Shadow Swap & HTTP QUERY)
* **Что исправляем:**
  1. Риск неполного создания или сбоя кэша из-за таймаута/заморозки фонового Server Action потока на Vercel (переносим на Upstash QStash).
  2. Исчезновение данных из CRM во время пересчета кэша (период между `DELETE` и `INSERT` переводим на атомарный Shadow Swap).
  3. Проблему передачи громоздких фильтров и плохой кэшируемости GET-запросов на клиенте (переводим на метод `QUERY` с поддержкой туннелирования `POST` + `X-HTTP-Method-Override`).
  4. Высокий расход ресурсов и сетевой payload при подсчете агрегированных сумм (переносим агрегации на SQL в СУБД).
* **Как тестируем на пользователях:**
  * Имитируем фоновый запуск обновления кэша во время активной работы 5+ менеджеров, проверяя непрерывность отображения данных (0 секунд пустого экрана).
  * Проверяем обработку метода `QUERY` и заголовка `X-HTTP-Method-Override` на наличие 403/405 блокировок у пользователей под корпоративными VPN/WAF.
  * Анализируем показатели времени отклика БД и объем передаваемого JSON-payload на панели QA Debug при сложных мульти-фильтрациях.
* **Когда переходим дальше:**
  * Полное отсутствие просадок отображения данных во время ребилда.
  * 100% завершенных задач ребилда кэша в логах QStash/Vercel.
  * Время ответа на запросы поиска и фильтрации лидов снижено до <150ms.


### v1.2: Оптимизация бэкенда и БД
* **Что исправляем:**
  1. Огромный Payload при фильтрации (выгрузка всех записей кэша в JS ради подсчета сумм).
  2. Перерасход ОЗУ при обработке всего непагинированного трафика clicks.
  3. Пиковые нагрузки на пул соединений Postgres при старте кэширования.
* **Как тестируем на пользователях:**
  * Проверяем скорость загрузки данных в CRM при выборе больших интервалов дат (30+ дней).
  * Измеряем параметры `dbFetchMs` и `payloadSizeKb` в логах производительности на панели QA Debug.
* **Когда переходим дальше:**
  * Payload ответа Server Action уменьшен до < 100 КБ для любого фильтра.
  * Время отклика `dbFetchMs` стабильно ниже 250ms под нагрузкой.

### v1.3: Оптимизация UI и UX
* **Что исправляем:**
  1. Медленный парсинг и компиляция JS-бандла браузером из-за монолита `LeadsDashboard.tsx` (240+ КБ).
  2. Торможение интерфейса (input lag) при вводе в строку поиска.
  3. Лишние перерисовки неактивных вкладок при смене фильтров.
* **Как тестируем на пользователях:**
  * Замеры метрик INP (Interaction to Next Paint) и CLS через Google Lighthouse / Chrome DevTools.
  * Тестирование скорости открытия страниц CRM со слабых мобильных устройств.
* **Когда переходим дальше:**
  * Оценка Performance в Lighthouse на мобильных > 85.
  * Задержка ввода (input lag) при фильтрации не превышает 50ms.

### v1.4: Database-Level Aggregation & Real-time Synchronization
* **Что исправляем:**
  1. Эффект "обмана данных" (Fake Data): менеджер меняет статус лида, но из-за фонового обновления кэша (DSU clustering) изменения появляются в дашборде только спустя минуты.
  2. "Кринж" In-Memory вычислений: скачивание всех десятков тысяч строк в Node.js для расчёта Leaderboard и Global Hub, что приводит к гигантскому расходу RAM и задержкам ответа до 10-20 секунд.
  3. Медленное построение кэша: алгоритм DSU на клиенте (Node.js) при росте базы становится "бутылочным горлышком".
* **Как тестируем на пользователях:**
  * Замер скорости отдачи дашборда для роли "Superman" (должно быть < 300ms, т.к. данные агрегируются на уровне СУБД).
  * Ручное изменение статуса лида (проверка, что данные появляются в таблице моментально за счёт Direct Cache Patch, без ожидания `rebuildProjectCache`).
* **Когда переходим дальше:**
  * Переход на SQL RPC (`get_superman_summary`, `get_producer_leaderboard`) завершён.
  * Прямая синхронизация `unified_orders` -> `crm_leads_cache` без промежуточного пересчета графа реализована.
