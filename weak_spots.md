# Анализ слабых мест и проблем производительности (B&W CRM)

## 1. Проблемы производительности (Performance Bottlenecks)

### Агрегация данных на стороне Node.js (Client-side/Server-action JS aggregation)
* В `getUnifiedCRMData` (`src/app/admin/actions.ts`) запрос `aggQuery` выгребает все записи кэша для проекта целиком.
* Подсчет сумм выручки по валютам (`usdCourseRevenue`, `uahCourseRevenue` и т.д.) выполняется через `reduce()` в JS-коде. При росте базы лидов это приводит к огромным объемам передаваемых данных (Network Payload) и нагрузке на CPU сервера.

### Загрузка нестраничного трафика (Unpaginated traffic fetching)
* Получение кликов трафика для проекта выполняется через `adminSupabase.from("traffic_clicks").select("*")` без ограничений `LIMIT` или пагинации.
* Фильтрация по датам происходит в коде через `.filter()`. При высоком трафике это приведет к перерасходу оперативной памяти и падению Serverless-функции по таймауту.

### Риск обрыва фонового обновления кэша в Vercel Serverless
* В `actions.ts` при необходимости пересчета кэша запускается `rebuildProjectCache(...)` асинхронно без ожидания (`.catch(...)`).
* В среде Vercel Serverless выполнение функции замораживается или прерывается сразу после отправки HTTP-ответа, из-за чего фоновый процесс пересчета кэша может быть завершен аварийно.

### Пиковая нагрузка на пул соединений в `fetchAllParallel`
* Функция `fetchAllParallel` в `crmCache.ts` запускает неограниченный `Promise.all` для параллельного скачивания всех страниц по 1000 записей. При больших объемах данных это создает резкий всплеск одновременных запросов к Supabase/PostgREST и может исчерпать пул соединений БД.

### Монолитный клиентский компонент `LeadsDashboard.tsx`
* Файл `LeadsDashboard.tsx` весит более 240 КБ и содержит более 4400 строк кода.
* Внутри сосредоточено управление всеми вкладками (Канбан, Сквозная аналитика, База лидов, Анкетирование, Настройки доступа, Генератор платежных кнопок).
* Любое изменение локального состояния (например, ввод текста в поиск или выбор фильтра) вызывает повторный рендеринг тяжелого дерева компонентов.

## 2. Логические ошибки и надежность данных (Reliability & Logic Issues)

### Неатомарное обновление кэша (Race Condition / Dirty Read)
* Метод `rebuildProjectCache` сначала выполняет `.delete().eq("project_id", projectId)`, а затем порциями по 500 записей делает `.insert(...)`.
* Запросы выполняются последовательно через обычный API клиент. Транзакция на уровне базы данных отсутствует.
* В случае ошибки во время вставки (например, на 2-й порции) или при чтении данных во время пересчета CRM покажет пустую базу или поврежденный кэш.

### Избыточное объединение лидов в DSU (False Positive Clustering)
* Алгоритм DSU объединяет записи по совпадению телефона, Telegram, Email или UUID.
* Если в системе присутствуют пустые, некорректные или шаблонные значения (например, тестовые телефоны вроде "1111111", "0000000", одинаковые некорректные telegram-юзернеймы), DSU схлопнет множество не связанных между собой клиентов в одну группу.

### Отсутствие `onConflict` в кэшировании расходов (Meta Spend Sync)
* В роуте `sync-spend/route.ts` метод `.upsert(chunk)` вызывается для таблицы `daily_traffic_and_costs` без указания параметра `onConflict`.
* Поскольку первичный ключ `id` (UUID) отсутствует в передаваемых данных, PostgREST может попытаться выполнить вставку и вызвать ошибку дублирования уникального индекса `unique_daily_spend`.

### Дублирование транзакций без `order_id`
* В `crmCache.ts` при группировке заказов для исключения дубликатов используется ключ `${orderId}_${item.project_id}`.
* Если у записи отсутствует `order_id`, в качестве резерва берется `item.id` (уникальный UUID строки). Из-за этого дублирующиеся записи по одной и той же транзакции (но с разными UUID строк) не склеятся и будут просуммированы повторно.
